AKG-Agents v2 深度介绍

一份基于 14 天逆向阅读（2026-05-02 → 2026-05-17）+ 仓库内代码证据的综合介绍 · 适用对象：想 90 分钟内建立 AKG-Agents 完整心智模型的资深开发者

TL;DR

AKG-Agents 是华为 MindSpore SIG 维护的 Multi-Agent GPU/NPU kernel 自动生成框架。3 个不可替代的特征：

  1. 昇腾 NPU 主战场 + CUDA 兼容—34 Ascend SKU + 4 CUDA SKU
  2. 三种优化策略并列—UCB (Adaptive Search) / 进化 (Evolve) / 长 ReAct (AutoResearch)
  3. 显式 Skill 系统 + 自动进化—SKILL.md + SemVer + SkillEvolution 4 模式

战略位置：在「AKG / AutoKernel / KernelAgent / AKO4ALL」四家 LLM kernel gen 框架里，AKG 工程最重、原创最多（trace / replay_guard / skill-evo / 设备池 / 服务化），但 NPU 是主战场，CUDA 是兼容性产物。

3. Agent 体系（core_v2/agents）

3.1 三层 Agent 继承

BaseAgent (单次 LLM 调用接口)
    ↓
ReActAgent (节点优先 6 步循环)
    ↓
领域子类 (KernelGen / Designer / Conductor / KernelAgent ...)

3.2 ReAct 节点优先 6 步循环

1. create_node          (先开 trace 节点)
2. inject cur_path      (注入上下文)
3. LLM call             (think)
4. parse                (act 解析)
5. exec tool            (observe)
6. save + update        (持久化 + 更新 state)

与教科书 ReAct 的差异：节点先于 LLM 创建，所以即使 LLM 崩溃也有 trace 可断点续跑。

3.3 AgentRegistry：Agent-as-Tool 自动转换

  • 所有有 TOOL_NAME 的注册 Agent 被 ReActAgent._load_agent_registry() 自动转成 OpenAI function calling 工具
  • multi-agent 通信走 LLM 路由而非 import
  • scope=set() 表示全局可见；类级单例，import 即注册

4. Skill 系统（core_v2/skill）

4.1 SKILL.md 数据模型

4+1 层结构：category（11 类）/ group（5 组）/ metadata（自定义）/ version（SemVer）+ structure（child / default / exclusive）

4.2 SkillSelector 两阶段筛选

阶段 1: 粗筛（4 类 filter AND 组合）
   metadata_matcher + category_filter (include/exclude)
   三层 fail-open: context 没指定 / skill 无 metadata / 字段空都放行
   支持 "all" 通配符 + 逗号分隔多值

阶段 2: LLM 精筛（用户自定义模板，v2 无默认模板）
   4 级 fallback: 直接 json / ```json 块 / 通用代码块 / 正则花括号
   失败退回粗筛结果

5. Workflow + Trace

5.1 LangGraph 三层抽象

  • BaseState — Reducer 语法 Annotated[List[str], add] 让多节点写同字段自动 append
  • Workflow — StateGraph + Conditional Edge + Checkpointer 断点续跑
  • 路由限制 — get_illegal_agents 防 LLM 死循环（默认连续 3 次同 Agent）

5.2 Trace 系统在 filesystem/

  • 14 种数据模型 + 多分叉 / 合并 / 行级 blame
  • 持久化：~/.akg/tasks/{task_id}/nodes/{node_id}/{state,result,thinking,code,logs,system_prompts}.json
  • CoW 代码快照：每次代码修改产生一份新拷贝，可回到任意节点 diff

5.3 replay_guard：multi-agent 回归测试守卫

  • 200 行代码，与 LLM cache 共用 JSON 文件（_node_snapshots key 隔离）
  • 专抓"LLM 给同样输出但节点解析出不同结果"这种隐蔽 bug
  • CI 必有 / better_kernels 应整体抄过来

    "tool": "do_something",

6.3 LLM 层

  • 3 档 model_level：complex / standard / fast，单次调用粒度切换
  • 双协议（OpenAI / Anthropic）+ extra_body 透传 reasoning 参数
  • cache/replay_catalog 跨会话录制/重放 LLM 调用 — 研究复现 / 调试 / CI 神器

7. op/ 场景层：四大 Agent

KernelAgent (顶层 ReActAgent)
   │
   ├─→ KernelConductor (单次 LLM，调度)
   │
   ├─→ KernelDesigner  (单次 LLM，算子规格→tile/调度方案)
   │
   └─→ KernelGen      (单次 LLM，方案→DSL 代码)
            │
            └─→ stage = initial / debug / optimize / pure-modify
                       (由输入字段决定)

KernelGen 三层 skill 分阶段加载：L0 always + L1 LLM 选 guide/example + L2 case；首次选择缓存到 _initial_selection_cache 后续阶段复用。

8. 算子生成全流程

LLM 生成 code
   ↓
CodeChecker      (6 步静态检查，YAML 驱动，不进 GPU)
   ↓
KernelVerifier   (三 adapter: backend × dsl × framework 正交分解 + Jinja2)
   ↓
Profiler         (profiler_npu warmup+active，L2 清除按 dsl 选)
   ↓
Solar Roofline   (roofline_speedup = roofline_time / gen_time)
   ↓
error_message → 拼进下次 KernelGen prompt → debug stage

9. 三层搜索：AKG 真正的差异化

9.1 短周期 — Adaptive Search (UCB)

单算子作用域：(op_name, task_desc) 绑定，存储 ~/akg_agents_adaptive_search/<op_name>/<ts>/。

偏离教科书 UCB实现目的
动态 arm 集合每轮重 rank适应新发现的 skill 组合
rank-based Qcontroller.py让 c=√2 跨算子可移植
batch 全失败回滚decrement_selection不浪费失败的探索预算
随机扰动 +Uniform(-r,+r)UCB1 公式后弥补 simple regret 弱点

9.2 中周期 — Evolve

双轨：v1 物理岛屿 GA (rounds × islands × parallel_num) vs v2 软岛屿外挂 (parent_id 谱系森林承担岛屿语义)。

v2 三个 Solver：S1 LineageAwareParentSelector（两阶段层间-层内）/ S2 LineageBoltzmannInspirationSelector（Boltzmann 采样 + 跨谱系迁移）/ S3 MultiDimensionalConvergenceDetector（EXPLORING→WATCHING→STOPPED 三状态机）。

三处温度 tau / c_local / T 默认 1.5→0.5 共用 progress 指数退火；WATCHING 时 S3 反向发 StrategySignal 主动升温 — FunSearch / AlphaEvolve 都没有的工程原创。

9.3 长周期 — AutoResearch

5 步 turn 循环：上下文护栏 → skill 注入 → LLM → tool 分派 → post-eval 钩子。预算 max_turns = max_rounds × 8。

三大原创：

  1. 每产物单一写方：plan.md→FeedbackBuilder / session.json→SessionStore / messages→ConversationBuffer
  2. SkillBuilder 无终态：tier 0/1/2 三级生命周期
  3. 三层 compact：micro / auto / force_rebuild + post-compact 5 marker buffer

10. 多 DSL + CUDA→Ascend 迁移

10.1 7 DSL 统一抽象

dsl_to_dir_key：triton_ascend → triton-ascend。每 DSL 自带 fundamentals/guides/examples/cases/evolved-* 子目录，只 triton-ascend 全齐（CUDA 侧只 guides + 少量 evolved）。

10.2 CUDA→Ascend 迁移不是翻译

Phase 1 (CUDA Worker)
   generate_reference_with_profile
        ↓
   .pt bytes + gpu_kernel_time_us

Phase 2 (NPU 优化)
   create_cross_platform_config 注入：
      config['use_reference_data']
      config['reference_data']
      config['gpu_kernel_time_us']
   复用 Day 10 evolve() 主循环但跳过本地 ref
        ↓
   extract_best_npu_kernel_time
   calculate_cross_platform_speedup → GPU/NPU 比率

Phase 2 KernelGen 完全不知道 Phase 1 用过 triton_cuda skill — NPU 侧不被 CUDA 习惯污染。

14.3 evolved 层是真护城河（举例）

# 错误：直接使用 tl.tanh
result = 0.5 * x * (1.0 + tl.tanh(inner))

# 正确：通过 libdevice 调用
result = 0.5 * x * (1.0 + tl.extra.cuda.libdevice.tanh(inner))

这种"具体 failure mode → fix"配对是 LLM 训练数据里罕见的，必须真跑过 SkillEvolution.search_log 才得到。
