1. 为什么需要复测

AKG-Agents v2 的 verifier 来自昇腾 NPU 工具链，默认容差远宽于 KernelBench 标准。

1.1 AKG verifier 容差

源码：experiments/frameworks/akg/akg_agents/python/akg_agents/op/verifier/adapters/framework/torch.py:285-394

def get_limit(data_type):
    if data_type == torch.float16:   return 0.004   # 0.4%
    elif data_type == torch.bfloat16: return 0.03    # 3%
    elif data_type == torch.int8:     return 0.01    # 1%
    else:                             return 0.02    # fp32 默认 2%

# 比较逻辑（简化）
relative_error = torch.where(abs_ref > eps, abs_diff / abs_ref, abs_diff)
err_cnt   = (relative_error > limit).sum().item()
limit_cnt = int(finite_count * limit)        # 允许 limit 比例的元素超阈值
if err_cnt > limit_cnt:
    raise AssertionError                     # 否则视为通过

也就是说，AKG 对 fp32 的判定是：

  • 每个元素允许 2% 相对误差，并且
  • 允许 2% 的元素超过该阈值。

1.2 Harness（bench.py / KernelBench）容差

torch.allclose(out_ref, out_test, atol=1e-4, rtol=1e-4)
# 所有元素必须满足；任意元素超阈值即 FAIL

两者差距：在 fp32 输出量级 ~1e3 的 matmul 上，AKG 允许约 20.0 的逐元素差，harness 只允许约 0.1。~200× 差距。这就是为什么 P1 matmul（max_diff=0.75，输出量级 1024，相对 ~0.07%）AKG 判 PASS、harness 判 FAIL。

这不是 bug：AKG 面向 NPU 混合精度场景设计，宽容差有助于把 fp16/bf16 加速器纳入相同框架。但比较多个 GPU 框架的"正确率"时必须用同把尺子。

6. 影响：跨框架对比的正确做法

之前的对比表（reports/l1/）中 AKO4ALL=77% / AutoKernel=42% / KernelAgent ≈ 60% 都是 harness 同标准下的 PASS 率。要把 AKG 放到同一张表，只能用本报告的 16/100 = 16%，而不是 AKG self-report 的 65%。

⚠️ 任何在 paper / 内部分享中引用 "AKG-Agents 通过 65 题" 的数字，都必须明确标注为 AKG-internal verifier（fp32 2% relative + 2% population），并附本报告作为 cross-framework 对照点。
