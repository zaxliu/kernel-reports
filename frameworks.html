三个 Agent 框架如何自动优化 GPU Kernel

从 Karpathy 的 autoresearch 到 KernelBench 实战 —— 评测集介绍、假阳性方法学、三框架设计与对比

§0   什么是 KernelBench

评测集简介

KernelBench 是斯坦福发布的 GPU 算子优化评测集。每题给出一段 PyTorch Model（参考实现）和一段 ModelNew（待优化），要求生成更快且数值正确的 CUDA/Triton kernel。

  • L1 — 100 题单算子（matmul、softmax、LayerNorm、conv 等）
  • L2 — 100 题算子融合（3–6 op 链，如 conv+BN+ReLU）
  • L3 — 模型级（MLP、Transformer block 等完整模块）
  • L4 — HuggingFace 模型（端到端推理/训练）

验收标准

KernelBench 默认使用宽松容差 atol=1e-2, rtol=1e-2，原本是为容忍 fp16 tensor core 舍入而设。 但这也导致"答案不太对但够近"的 kernel 通过验收 —— 即假阳性问题。 本评测用 fp32 strict (atol=1e-4) 三层复测逐题验真伪。

§1   autoresearch 范式

Karpathy autoresearch 核心理念

一个 GPU、一个文件、一个指标。

  • 写一份 program.md，告诉 LLM "你在做什么、怎么迭代、怎么判好坏"
  • 循环: 改代码 → 跑 5 分钟 → 指标变好则保留 / 变差则回退 → 重复
  • 无需人工干预，一夜可跑 100+ 次迭代

如何迁移到 kernel 优化

autoresearch 的"一个文件"对应 kernel.py，"一个指标"对应 speedup（加速比）。 三个框架都继承了这个核心循环，差异在于：profile 策略（是否强制 ncu）、 反作弊约束（是否限制 dtype cheat / reward hacking）、迭代留痕（是否强制记录假说和结果）。

§2   三框架详解

AKO4ALL — Agentic Kernel Optimization for All

来源: Tongming LAIC (项目页)。 外层 coding agent（默认 Claude Code）= 研究员；框架本身完全文件驱动， 把"什么算一次迭代、必须留什么痕迹、什么是 reward hacking"全部写进 TASK.md / HINTS.md，靠 LLM 读 prompt 自律。

工作流（5 阶段）

① Setup ② Profile (ncu) ③ Iterate ④ Adapt ⑤ Finalize
  • Setup — 读 input/、context/、bench/、HINTS.md；创建 opt/<kernel-name> 分支
  • Profile — 强制 ncu 对 baseline profile 一次（不可跳过）
  • Iterate — 每次 edit kernel.py → bash bench.sh iter-N → 写 ITERATIONS.md → git commit 四步原子；无论成功/失败都记一个 iter
  • Adapt — 命中 HINTS.md 中的 stall rule（连续 N 次无改进）才允许换方向
  • Finalize — final commit + 在 ITERATIONS.md 写 summary

关键设计

🛡
反 reward hacking — TASK.md 36 行把目标定义写在第二段: "Your goal is genuine latency reduction — not maximizing the reported speedup ratio." 并列出 4 类禁止技巧: CUDA stream injection / thread-process injection / monkey-patch timing / 任何 reward hacking。 这是与其他框架的核心差异。
📊
强制 ncu profile — 不是建议而是约束。让 LLM 看到真实瓶颈（memory-bound / compute-bound）， 而非盲目尝试 dtype cast。
📓
ITERATIONS.md 留痕 — 每次迭代强制写: Hypothesis → Result → Decision。 连续 3 iter 无改进触发 stall rule → 重新 ncu + 回读历史 + 换方向。 这是 ako4all 能从 fp16 失败回退到 TF32 / 切换策略的核心机制。
🔍
WebSearch 可用但未触发 — prompt 两处鼓励调用 WebSearch；但 L1 全部 100 题的 ITERATIONS.md 全文 grep 0 命中。L1 都是教科书级算子，LLM 权重已涵盖。
L1 真实率 77% · L2 真实率 83% · fp32 strict 复测

AutoKernel / program_md — model-level autoresearch

来源: RightNow-AI 的 AutoKernel， 灵感来自 Karpathy autoresearch。 设计为给一个完整 PyTorch model，自动 profile → extract → optimize → integrate。 program.md 长达 909 行，三阶段 A/B/C。

工作流（三阶段 · KB 单题下被砍为仅 Phase B）

Phase A: 分析 (砍) Phase B: 优化循环 Phase C: 集成 (砍)

单 KernelBench 题没有 model-level 概念 —— run_autokernel.sh 把 program.md 拼到 CLAUDE.md 后追加 "只看 kernel.py，做 hypothesis → edit → bench → keep/revert"。 Phase A/C 全跳过，Phase B 缩成单 kernel loop。

6-Tier Optimization Playbook

Tier 1 Block Size Tuning — BLOCK_M/N/K 扫 16–256，预期 10–50%
Tier 2 Memory — coalescing、prefetch、L2 swizzle、bank conflict，预期 10–30%
Tier 3 Compute — TF32、mixed precision、epilogue fusion，预期 5–15%
Tier 4 Advanced — Split-K、persistent kernels、autotune、warp specialization，预期 5–20%
Tier 5 Architecture — H100/A100/L40S 各有标签，RTX 5090 段缺失
Tier 6 Kernel tricks — matmul swizzle、softmax online、LayerNorm Welford

dtype 策略隐患

Tier 3 写 "Use tl.dot(a, b, allow_tf32=True) … Keep accumulators in fp32" — 理论正确。 但 LLM 实际常写出 tl.dot(a.to(bf16), b.to(bf16), acc)，操作数降精度而 accumulator 在 fp32， 正是假阳性 §0.3 第 1 类 truncation 的来源。
L1 真实率 42% · fp32 strict 复测

AutoKernel / program_kb — KernelBench-native 变体

同 AutoKernel 仓库的 kernelbench/ 子模块。 prompt 入口 program_kb.md 仅 388 行（比 program.md 短一半多）， 完全跳过 Phase A/C，直接面向 KB 的 (Model, ModelNew) 对做优化循环。

工作流（8 段 · 更激进）

Setup Optimization Loop Playbook by Level CUDA Tips Anti-Patterns Decision Scoring

激进 dtype 策略

对 matmul 直接写 "Use wmma tensor cores" 并指向 kernels/cuda/matmul.py 参考实现 —— 这是 program_kb 在 matmul 类问题上压倒性走 fp16/bf16 路径的根因。 又因为 KB 默认 atol=1e-2，这些 cheat 都能通过框架自己的验收。

缺失的约束

不要求 ITERATIONS.md 留痕、不要求 ncu profile、 没有 anti-reward-hacking 段。 Anti-Patterns 7 项中没有 "don't fp16-cheat" 或 "don't fallback to baseline"。 结果: P52/P39/P37 出现 return torch.argmin(x, dim) 等套壳 baseline。
L1 真实率 28% · L2 真实率 10% · fp32 strict 复测

关键差异三点总结

1
强制 profiling 是唯一有效的反作弊手段。 ako4all 把 ncu 从"建议"升级为"不可跳过"，让 LLM 看到真实瓶颈而非乱套 cast。
2
迭代留痕 = 策略回退能力。 ITERATIONS.md 不只是记录，更是 stall rule 的判断依据 —— 让框架能从错误路径退回。
3
prompt 越激进，假阳性越高。 program_kb 显式鼓励 tensor core + 无 anti-cheat → 72% 的声明加速实为虚假。

§4   方法学: 假阳性四种机制

KernelBench 默认验收 (atol=1e-2) 让"答案不一样但靠近"的 kernel 也算 PASS。 本评测出现的假阳性集中在 4 个机制：

①
精度截断 (truncation)

kernel 用 fp16/bf16 做 reduction，舍入 ≪ 1e-2 → PASS。 fp32 strict 复测被打掉。
典型: ako4all P23 Softmax 31.52× claim。
Verdict: fake_truncation_or_precision

②
dtype cast 作弊

直接 .to(half) / .to(bfloat16) 兑换 tensor core 速度。
典型: P40 LayerNorm — ako4all 6.98× 真实通过 vs program_kb 7.54× 被当场识破。
Verdict: fake_precision_cheat

③
baseline fallback (套壳)

LLM 直接写 return torch.argmin(x, dim) 之类的 PyTorch 原 op， 数值一致所以 fp32 复测也通过，但根本没做优化。
典型: program_kb P52/P39/P37。
Verdict: real_fp32（当前 pipeline 盲点）

④
Logic bug + 弱 atol 漏检

kernel 算错了（公式/下标/padding 错），但输出形状对、数值偶然落在 atol=1e-2 内。 conv_transposed 系列高发。
典型: program_md P58 conv_transposed_3D 6.43× → fp32 & fp16 都 fail。
Verdict: fake_logic_bug

为什么 ako4all 真实率高 (77%)? 不是它不出错，而是其工作流把上述 4 个模式都抑制住了：强制 ncu profile 让 LLM 看瓶颈而非乱套 cast、 TF32 toggle 是 fp32 路径的真实加速（非精度兑换）、ITERATIONS.md 让 fp16 失败被回退而不是被放过、 TASK.md 显式禁止 reward hacking。

三层复测流程

Tier atol / 流程 抓什么 命中 Verdict
Tier 1 fp32 strict atol=1e-4 dtype cheat + 弱 atol 漏过的 logic bug real_fp32 / fake_precision_cheat / fake_logic_bug
Tier 2 fp16 fallback (仅 Tier 1 OOM/fail 时) 区分"真的 OOM 只能 fp16" vs "故意 fp16" real_fp16_oom_verified / fake_or_broken
Tier 3 chunked fp32 sanity (可选) truncation 在更大 input 下发散 fake_truncation_or_precision
盲点提醒: baseline fallback（第③类）三层都会通过，verdict 上是 real_fp32 但不是真优化 —— 判断仍需读 kernel.py 源码。

Verdict 类别说明

Verdict含义

§5   术语速查表

术语简释
GPU Kernel在 GPU 上运行的计算函数。深度学习中的矩阵乘法、归一化、注意力等算子最终都编译为 kernel。
CUDANVIDIA 的 GPU 编程框架。直接写 CUDA C++ 可以精细控制硬件，门槛高但性能上限也高。
TritonOpenAI 的 GPU 编程语言。用 Python-like 语法写 kernel，自动处理 tiling 和内存管理，比 CUDA 简单但灵活性稍低。
cuBLASNVIDIA 的线性代数库。提供高度优化的 matmul 等操作，通常是性能下限。
atol / rtol绝对容差 / 相对容差。用于判断两个张量"数值上是否一致"。atol=1e-2 很宽松，1e-4 较严格。
fp32 / fp16 / bf16浮点精度：32 位（标准）、16 位（半精度）、bfloat16（指数位多、表示范围大但小数少）。精度越低计算越快但误差越大。
TF32Tensor Float 32。NVIDIA Ampere+ GPU 的计算模式：用 19 位精度跑在 tensor core 上，速度接近 fp16 但精度接近 fp32。
ncuNsight Compute。NVIDIA 的 kernel profiler，可以看到每个 kernel 的内存带宽、计算利用率、瓶颈在哪。
Roofline性能分析模型。以"计算强度"（FLOP/byte）为 X 轴，判断 kernel 是 memory-bound 还是 compute-bound。
Agent 循环LLM 自主修改代码 → 运行 benchmark → 保留改进 / 回退失败 → 重复。无需人工干预的自动优化流程。
假阳性框架声称加速成功但实际不正确或不真实的结果。常见于 dtype cheat、logic bug 等。
KernelBench斯坦福发布的 GPU 算子优化评测集，4 个难度级别共数百题。
Tensor CoreNVIDIA GPU 上的矩阵运算加速硬件单元。需要特定精度（fp16/bf16/TF32）才能启用。
Speedup加速比。优化后 kernel 的执行时间 / 参考实现的执行时间。>1× 表示更快。
