0. 方法学 · 为什么会有假阳性

口径前提 KernelBench 默认验收 atol=1e-2, rtol=1e-2 是为了容忍 fp16 tensor core 舍入而留的, 结果让"答案不一样但靠近"的 kernel 也算 PASS。 本评测出现的假阳性集中在下面 4 个机制。

① 精度截断 (truncation)

kernel 用 fp16/bf16 做 reduction,舍入 ≪ 1e-2 → PASS; fp32 strict (atol=1e-4) 复测被打掉。

L2 典型: ako4all P10 fp16 ConvTranspose + Triton mean+tanh、 program_md P64 bf16 GEMM。
verdict: fake_truncation_or_precision

② dtype cast 作弊

直接 .to(half) / .to(bfloat16) 兑换 tensor core 速度。

L2 教科书例: P9 Matmul-Sub-Mul-ReLU — program_kb 9.49× claim (__nv_bfloat16 整链, fp32 strict 失败) vs ako4all 2.54× real (fp16 GEMM + fp32 epilogue)。
verdict: fake_precision_cheat

③ baseline_fallback (1.00× 套壳)

LLM 直接写 return self.linear(x) 之类 PyTorch 原 op, verdict 上仍记为 real_fp32 但没做优化。

L2 典型: ako4all P19/P66 verified ≈ 1.00× 都是这类。
verdict 仍记 real_fp32 — 当前 pipeline 的盲点。

④ Logic bug + 弱 atol 漏检

kernel 实际算错 (公式/下标/padding 错),输出形状对、 数值偶然落在 atol=1e-2 内。最阴险 — 不是套壳也不是 cheat。

L2 ConvTranspose 系列高发 — program_kb 6/10 都是这类, 典型: P5 v20_final_verify 200+ 行 fused CUDA kernel boundary 错。
verdict: fake_logic_bug

L2 vs L1 的真实率反转点 L1 ako4all 77% → L2 83% 反而升 (L2 多 op chain 给了 LLM 更大"数学塌缩"算法空间); L1 program_kb 28% → L2 10% 坍塌 (ConvTranspose fused CUDA kernel 几乎必出 boundary bug)。

verdict 与三层复测的对应

Tieratol / 流程抓什么命中 verdict
1 fp32 strict atol=1e-4 dtype cheat + 弱 atol 漏过的 logic bug real_fp32 / fake_precision_cheat / fake_logic_bug
2 fp16 fallback (仅 Tier 1 OOM/fail 时) 区分"真的 OOM 只能 fp16"vs"故意 fp16" real_fp16_oom_verified / fake_or_broken
3 chunked fp32 sanity (可选) truncation 在更大 input 下发散 fake_truncation_or_precision
盲点提醒 baseline_fallback (机制 ③) 三层都会通过, verdict 上是 real_fp32 但不是真优化;判断仍需读 kernel.py。

ako4all · AKO4ALL — Agentic Kernel Optimization for All

Tongming LAIC · project page · 同团队还有 AKO4FIB (针对 flashinfer-bench operators)

外层 coding agent (默认 Claude Code，理论上换任意 IDE-style agent 也能用) = 研究员； 框架本身完全 文件驱动，不依赖任何特定 LLM SDK。把"什么算一次迭代、必须留什么痕迹、 什么是 reward hacking"全部写进 TASK.md/HINTS.md， 靠 LLM 读 prompt 自律。

工作流 (5 阶段)

  • Setup — 读 input//context//bench//HINTS.md， 确认输入形状可得；create opt/<kernel-name> 分支并拷贝 kernel 到 solution/。
  • Profile — 强制 ncu 对 baseline profile 一次 (不可跳过)。
  • Iterate — 每次 edit → bash scripts/bench.sh iter-N → 写 ITERATIONS.md → git commit "[iter N] ..." 强制四步；任何修改 + 一次 bench = 一个 iter，不分成功/失败。
  • Adapt — 命中 HINTS.md stall rule (连续 N 次无改进) 才允许换方向。
  • Finalize — final commit + ITERATIONS.md 写 summary。

目录结构 (本评测下)

AKO4ALL/
├── TASK.md              ← agent 启动 prompt (强制 iteration protocol + 反 reward hacking)
├── HINTS.md             ← stall rules / 调参约束
├── input/               ← 用户起始 kernel
├── bench/kernelbench/   ← 内置 KernelBench evaluator (bench.py / GUIDE.md)
├── context/             ← 可选参考材料
├── bench-wrapper.sh     ← scripts/bench.sh 模板 (tee 到 trajectory)
├── solution/            ← agent 创建，实际优化的 kernel
└── ITERATIONS.md        ★ 每次 bench 后强制留痕，真实率关键证据

关键文件

TASK.md 36 行 把行为约束塞进单 prompt：iteration protocol + 禁 stream injection / monkey-patch / timing 篡改
HINTS.md stall 触发条件 + 哪些 hint 可被覆盖
bench/kernelbench/bench.py KernelBench 单题 bench engine，5 阶段 correctness check，atol/rtol 可配
bench-wrapper.sh 把 bench 输出 tee 到 trajectory/iter-N/，所有历史可查
ITERATIONS.md 运行时生成 hypothesis-driven 留痕，是 ako4all 真实率高的关键证据来源
核心 playbook 解构 (TASK.md 36 行 / HINTS.md 10 行)
  • 核心 playbook 解构 (TASK.md 36 行 / HINTS.md 10 行):
    • 目标定义（Goal）: "Your goal is genuine latency reduction — not maximizing the reported speedup ratio." 紧接列出 4 类禁止技巧: CUDA stream injection 绕过 timing / thread-process injection / monkey-patch timing 或 bench 脚本 / 任何 reward hacking。这一段是与其他 3 个框架的核心差异：把"什么算正经优化"刻在 prompt 第二段。
    • Setup 5 步: ① 读 input//context//bench//HINTS.md → ② create opt/<kernel-name> 分支 → ③ 拷 kernel 进 solution/ → ④ 生成 bench.sh（替换 {{BENCH_COMMAND}}，自动 tee 到 _bench_output.txt）→ ⑤ 运行 baseline 验证环境 + 首次 [baseline] commit。
    • Iteration Protocol（3 步原子）: 每次 edit kernel.py → bash scripts/bench.sh iter-N → write ITERATIONS.md → git commit "[iter N] ..." 必须四步全做完才能开下一轮，无论这一轮是 improved / regression / failed。任何修改 + 一次 bench = 一个 iter。
    • Optimization 期约束: ① 必须用 ncu profile，不能瞎改 ② 必须参考 context//HINTS.md/历史 attempts/WebSearch ③ 必须遵守 HINTS.md 中的 stall rules。
    • HINTS.md Stall Rules: ① Iter 1 之前先 ncu profile baseline ② 连续 3 iter 无 improvement → 强制重新 ncu + WebSearch + 回读 ITERATIONS.md patterns + plan before continuing。这是 ako4all 在长 trajectory 中能从 fp16 失败回退到 TF32 / 切换策略的核心机制。
    • 未在 prompt 出现的东西: dtype cast 推荐、tensor core 推荐、specific BLOCK_SIZE 建议。整套 prompt 不告诉 LLM "怎么改"，只告诉它"必须留痕、必须 profile、必须诚实"。
    • WebSearch — 允许但实测未触发: prompt 在两处显式鼓励调用 WebSearch（TASK.md:25 列为可用信息源、HINTS.md:4 stall rule 建议 "use WebSearch for new ideas"）。本评测的外层 agent 是 Claude Code，自带 WebSearch + WebFetch 工具，技术上随时可调。但服务器上全部 100 题 ITERATIONS.md 全文 grep websearch / arxiv / developer.nvidia / github.com / stackoverflow → 0 命中；本地 3 个深度案例 (P40/P15/P14) 的 kernel.py 注释也无任何 URL/paper 引用。结合 ITERATIONS.md 强制写出 Hypothesis 的留痕约定（如果引了外部资料应该会留痕），可以认定ako4all 在 L1 评测中的全部优化策略都来自 LLM 权重内的领域记忆 + 当次 ncu profile + 自身 ITERATIONS.md 历史，WebSearch 始终是一个可用但未启用的备份信号。
    • 为什么 WebSearch 在 L1 上没被触发: ① L1 都是教科书级 single-op (matmul / softmax / layernorm / element-wise)，LLM 权重已涵盖；② stall rule 让 LLM 在 "重 ncu profile" 与 "WebSearch" 之间选了更便宜的前者；③ Claude Code 的 WebSearch 默认有用户权限确认，长批跑容易 silent skip。这条对 L2/L3 (fusion / 全架构) 或新硬件特性 (sm_120 TMA/WGMMA) 的题目预期会不同，是 follow-up 评测的观察点。

program_md · AutoKernel — model-level autoresearch

RightNow-AI · github · 灵感来自 Karpathy/autoresearch

给一个 完整 PyTorch model，自动 profile → extract → optimize → integrate； agent 只动一个 kernel.py，剩下用 orchestrate.py 按 Amdahl 决定下一个该改哪个 kernel。设计为夜间无人值守跑 ~320 experiments/8h。

工作流 (program.md ~909 行，三阶段)

  • Phase A — Model Analysis (与人交互): profile.py ncu profile，识别 top bottleneck，与人确认优化计划
  • Phase B — Multi-Kernel Optimization (全自治): 每个 bottleneck extract.py 切出 → 单 kernel 优化循环 (edit → bench → keep/revert) → 收敛后 orchestrate 下一个
  • Phase C — Integration & Verification: verify.py end-to-end 验证 + 最终 report
本评测下被砍掉: 单 KernelBench 单题没有 model-level 概念。 外层 run_autokernel.sh 把 program.md 拼到 CLAUDE.md 后追加 "profile/extract/orchestrate 不可用，只看 kernel.py"。 Phase A/C 全跳过，Phase B 缩成单 kernel loop。

目录结构

autokernel/
├── program.md           ★ 主 prompt (~900 行三阶段 A/B/C)
├── orchestrate.py       ← 三阶段编排 (Amdahl 决策)
├── profile.py           ← model 级 ncu profile (KB 单题模式被砍)
├── extract.py           ← 切出 bottleneck kernel (KB 单题模式被砍)
├── bench.py             ★ 单 kernel bench: 5-stage correctness + roofline
├── verify.py            ← end-to-end 验证
├── kernel.py            ★ agent 实际改的文件 (KB 单题模式仅此一个)
├── reference.py         ← golden
├── kernels/             ← 已优化 kernel 仓库 (reuse)
├── models/              ← demo (LLaMA / GPT-2 / BERT)
└── kernelbench/         ← program_kb 入口 (详见下一 tab)

关键文件

program.md 909 行 三阶段大 prompt；KB 单题模式只用 Phase B + Optimization Playbook + Decision Framework 三段
orchestrate.py Amdahl 决定"再优化此 kernel 边际收益 < threshold 就跳到下一个"
bench.py 5-stage 验收 (compile / forward / backward / shapes / numerical)，默认 atol=1e-2
kernel.py 包含 ModelNew (优化目标) 和 Model (reference)，KB 单题模式下唯一可编辑文件
核心 playbook 解构 (program.md 909 行 · 3 阶段 + 6-Tier Optimization Playbook)
  • 核心 playbook 解构（program.md 909 行）:
    • Phase A 10 步（A1-A10，KB 单题被砍）: 人交互、profile.py ncu profile、读 profile report、extract.py 切 kernel、出优化计划、人确认、create branch、读所有文件、verify env、init results.tsv。
    • Phase B 单 kernel 循环 8 子步（B4.1-B4.10）: ① Hypothesize → ② Edit kernel.py → ③ Commit → ④ Run → ⑤ Check Results → ⑥ Decide KEEP/REVERT → ⑦ Record with orchestrator → ⑧ Analyze roofline → ⑨ Log → ⑩ Check orchestrator。在 KB 单题模式下，⑦/⑧/⑩ 都退化（无 orchestrator）。
    • Phase C 5 步: end-to-end verify、handle 失败、diagnose、生成 final report、optional 分析。KB 单题模式跳过整 Phase C。
    • Optimization Playbook 6 Tier（共 ~130 行）:
      • Tier 1 Block Size Tuning: BLOCK_M/N/K 扫 16/32/64/128/256；num_warps/num_stages 作次级调参。预期 10-50% 收益。
      • Tier 2 Memory: coalescing、tl.prefetch 软流水、L2 swizzling、shared mem bank conflict。预期 10-30%。
      • Tier 3 Compute: TF32 (tl.dot(..., allow_tf32=True))、mixed precision、epilogue fusion、tl.where 替分支。预期 5-15%。
      • Tier 4 Advanced: Split-K、persistent kernels、@triton.autotune、warp specialization、register tiling。预期 5-20%。
      • Tier 5 Architecture-specific: H100/A100/L40S 各自标签，RTX 5090/Ada Lovelace 段缺失 — 这是 LLM 在 5090 上瞎猜 BLOCK_SIZE 的根因之一。
      • Tier 6 Kernel tricks: matmul swizzle / softmax 在线 / LayerNorm Welford。
    • Anti-Patterns (Things That Usually Do Not Work): 12 项含 "Tiny block sizes (BLOCK=8)" / "Excessive splits" 等。不含"don't fp16-cast"。
    • Decision Framework: when to move on（K 个 iter 无改进或边际收益 <X%）、when to revisit、when to radical change。
    • Mixed precision 的官方立场: Tier 3 明写 "Use tl.dot(a, b, allow_tf32=True) for matmul accumulation with TF32 inputs. Keep accumulators in fp32 for numerical stability. Cast results to output dtype only at the end." — 这是个理论正确的描述，但 LLM 实际写出来的常常是 tl.dot(a.to(bf16), b.to(bf16), acc) 把操作数也降精度，accumulator 在 fp32 但运算输入已经降精度，正是 §0.3 第 1 类 truncation。

program_kb · AutoKernel — KernelBench-native 变体

AutoKernel 仓库的 autokernel/kernelbench/ 子模块 · prompt 入口 program_kb.md (388 行，远比 program.md 紧凑)

既然单 KernelBench 问题没有 model-level extract 概念，不如完全跳过 Phase A/C， 直接面向 KB 的单 (Model, ModelNew) 对做 prompt-loop。

工作流 (program_kb.md 8 段)

关键文件

program_kb.md 388 行 比 program.md 短一半多；面向 KB 已知 atol=1e-2， 显式鼓励"fp16/bf16 cast 换 tensor core"作为常见策略 → 这是 fake_precision_cheat 集中在 program_kb 的根因之一

核心 playbook 解构 (program_kb.md 388 行 · 8 段 + Playbook by Problem Level)
  • 核心 playbook 解构（program_kb.md 388 行 / 8 段）:
    • Phase 1 Setup（1.1-1.4）: Fetch problems from HF / list / set up / read problem.
    • Phase 2 Optimization Loop（2.1-2.8）: Run baseline → Hypothesize → Edit kernel.py → Commit → Run → Parse results → Keep-or-revert → Repeat. 跟 program_md 的 B4 类似但更紧凑。
    • Optimization Playbook by Problem Level（不是 by tier 而是 by difficulty）:
      • L1 Single Operations: 对 element-wise 推 grid-stride loop + float4 vectorized；对 reduction 推 __shfl_down_sync + shared memory；对 matmul 直接写 "Use wmma tensor cores"，并指向 kernels/cuda/matmul.py 作为参考实现 — 这一句是 program_kb 在 matmul 类问题上压倒性走 fp16/bf16 路径的根因。
      • L2 Fusion: 识别 3-6 op chain，写单 kernel 消中间 mem。
      • L3 Architecture: profile 找 hot path + 选择性优化。
      • L4 HuggingFace: torch.compile baseline + 选择性替换。
    • CUDA C++ Tips: 用 kernels.cuda._compile.compile_cuda，dtype handling、多输出、参数处理。
    • Anti-Patterns（7 项）: ① 不改 reference.py ② 优先 correctness ③ 别过度优化 PyTorch 已最优 ④ 别忘 torch.no_grad() ⑤ 编译错误读消息 ⑥ 别假设 specific shape ⑦ 别破坏 ModelNew.__init__ 签名。没有"don't fp16-cheat"或"don't fallback to baseline"。这是 program_kb 出现 return A.half() @ B.half() (P14) / return torch.argmin(x, dim) (P52) 等套壳的 prompt 漏洞。
    • Decision Framework: ① 何时用 CUDA/Triton/PyTorch 的对照表（matmul → "wmma 或 Triton"）② 何时换题: speedup > 2.0× 稳定、15+ exp 无改进、操作已达硬件极限、ROI 太低。
    • Scoring: fast_p at 1.0/1.1/1.25/1.5/2.0/3.0/5.0×，目标 fast_1 > 0.80。
    • 缺失的强制约束: 不要求 ITERATIONS.md 留痕、不要求 ncu profile、没有 anti-reward-hacking 段。所以本评测看到 program_kb 多数题 trajectory 只有 expN_<note> 命名的目录 + 一份空 output.txt（只 final 那个 bench output 有内容）。

关键差异要点
  1. Profile-driven vs blind edit: 只有 ako4all 强制 ncu, 也是唯一能稳定看到 hypothesis-driven iteration 痕迹的框架。 AutoKernel 两 variant 在 KB 单题模式下几乎跳过 profile。
  2. dtype 默认策略决定真实率: AutoKernel 偏好 fp16/bf16 cast 换 tensor core,KB atol=1e-2 漏检 → fp32 strict 复测多被打掉(fake_precision_cheat 集中)。
  3. 迭代深度差距: ako4all 平均 18-25 iters,trajectory 里能看到 baseline → Triton fp16 失败 → 回退 TF32 → 局部 BLOCK 调参的收敛过程; AutoKernel 单一 final_* kernel 缺乏 hypothesis 演化痕迹。
  4. 结论: 在 KernelBench L2 / RTX 5090 赛道, ako4all 是唯一一个 fp32 strict 下仍能稳定产出真实加速的框架; 其他两个 (AutoKernel 双 variant) 的高 claim 多半在复测时被打掉。

4. 各框架策略画像

ako4all

  • 数学塌缩 / 代数等价: 识别 op chain 中的化简结构 — P18 整链塌缩到 matvec 拿到 92.76× real_fp32,L2 最大真实加速
  • fp32 IEEE Triton + tile-skip: P4 Mish×2 (1.30×) / P9 GEMM-Sub-Mul-ReLU (2.54×) / P20 ConvTranspose3d-Residual (2.91×)
  • cuDNN baseline + 轻量 epilogue 融合: 多数 Conv 类保留 cuDNN 主 kernel,只 fuse 后续 norm/activation
  • 退化策略: ConvTranspose 太复杂时回到 baseline (P11/P15 trajectory 30+ iter 后 final=baseline)
  • 平均 15-30 iters,hypothesis-driven 可见;长 trajectory 偶发 ITERATIONS.md 留痕断档 (§5.3 P10)

program_md

  • Triton autotune fused kernel: P9 (1.88× real)/ P10 (1.07× real) / P60 (1.24× real),100-150 行 kernel
  • Mixed precision tensor core: P72 显式 bf16 → fake_precision_cheat
  • Conv 类容易出 logic bug: P10/P11/P19/P34 (claim 73.20× 实际 verified=-1.0)
  • 仅 9 题样本,均带 exp<N>_<note> 命名,少 ITERATIONS.md 留痕

program_kb

ako4all P18 最终 kernel 全文要点

class Model(nn.Module):
    """
    Optimized: The last 4 ops are identity on (batch, 1) tensors.
    The matmul+sum is reduced to: x @ sum(W.T, dim=1) + bias.sum()
    = x @ W_sum + bias_sum, a matrix-vector product.
    """
    def __init__(self, in_features, out_features):
        super().__init__()
        self.linear = nn.Linear(in_features, out_features)
        with torch.no_grad():
            # ★ sum(x @ W.T, dim=1) = x @ sum(W.T, dim=1) = x @ W_sum
            self.register_buffer('W_sum', self.linear.weight.sum(dim=0))     # (in_features,)
            self.register_buffer('bias_sum', self.linear.bias.sum(dim=0, keepdim=True))

    def forward(self, x):
        # x @ W_sum + bias_sum  :  (batch, in_features) @ (in_features,) → (batch,)
        x = torch.matmul(x, self.W_sum) + self.bias_sum
        return x.unsqueeze(-1)  # (batch, 1)

为什么这样合法:

  1. Linear(x) → sum(dim=1): sum_j (x @ W^T)[i,j] = x[i] · sum_j(W^T[:,j]) = x[i] · W_sum，复杂度从 O(B · in · out) (matmul) + O(B · out) (sum) 塌缩到 O(B · in) 单次 matvec。out_features=8192 的代价完全消失。
  2. (B, 1) → max(dim=1): 单元素 max = 自身；(B,) → unsqueeze(-1) → (B,1)。
  3. avg_pool1d(kernel=1): 也是 identity。
  4. logsumexp(dim=1) on (B, 1): lse([x]) = log(exp(x)) = x，也是 identity。
  5. 再来一次 logsumexp(dim=1): 仍 identity。

第 1 步把 matmul 计算量从 1024 × 8192 × 8192 ≈ 6.9e10 FLOPs 降到 1024 × 8192 ≈ 8.4e6 FLOPs，理论加速比 8192×；fp32 cuBLAS GEMM 与 GEMV 的实际硬件利用率差异给出最终 92.76× 的"实测下界"。

为什么 program_md / program_kb 没拿这道题:

两 variant 在 L2 上的样本里都没选 P18 — 选题机制使然，并非"看了选输"。但即使选了，program_kb.md L2 Fusion 段建议是 "写单 fused kernel 消中间内存"，LLM 几乎不会去想"数学等价变换"：它会直接 fuse 出一个 5-pass CUDA C++ kernel 来串联五个 op，得到 ≤ 2× 的 kernel-fusion 收益，远低于算法塌缩的 92×。

为什么 ako4all 拿到了:

  1. TASK.md 不规定怎么改: prompt 第一段只说 "Your goal is genuine latency reduction"，第二段禁 reward hacking，不告诉 LLM "必须写 kernel"。当 LLM 看到 reference 的 op 链能数学塌缩时，"修改 forward 用一个等价的 cheap 写法" 是 prompt 允许的合法选项。
  2. ITERATIONS.md Hypothesis 段逼自查: iter-1 标题 "Mathematically fuse linear+sum, eliminate post-sum identity ops" 把"为什么这样改不破坏 correctness"明确写出来 — 这一步是 ako4all 把 LLM 的"代数推理"调出来的关键。
  3. HINTS.md 没要求"必须写 GPU kernel": stall rules 触发条件是"3 iter 无 improvement"，iter-1 直接 +92× 巨大幅度，根本不会触发 stall。

教训: L2 / L3 / 真实模型上的最大加速来源不是 kernel 调优而是对算子链的代数化简。这要求 prompt 必须给 LLM 足够的空间去"重新写 forward"，并强制留痕"为什么等价"。ako4all 这两点都做到了；program_md/program_kb 的 prompt 倾向用 "写一个 fused kernel" 收窄了 LLM 的搜索空间。

ako4all P9 最终 kernel 关键片段（iter-5, 2.54×）

class Model(nn.Module):
    def forward(self, x):
        # ★ FP16 tensor core matmul, FP32 elementwise (atol-safe for sub+mul+relu)
        w = self.linear.weight.to(torch.float16)
        x_fp16 = x.to(torch.float16)
        x = torch.matmul(x_fp16, w.t())
        if self.linear.bias is not None:
            x = x + self.linear.bias.to(torch.float16)
        x = x.to(torch.float32)
        x -= self.subtract_value          # ★ fp32 elementwise in-place
        x *= self.multiply_value
        return torch.relu_(x)

为什么 ako4all 这次 fp16 cast 没被打成 fake_precision_cheat：matmul output 立刻 cast 回 fp32，之后 subtract / multiply / relu 全 fp32。fp16 在 matmul 内的舍入误差被 epilogue fp32 计算"吸收" — 输出值域大、subtract+multiply 不放大相对误差、relu 把负值清零。verify_three_tier_l2 在 atol=1e-4 下 max_abs_error < 0.005，真实通过。这与下面 program_kb 把 sub+mul+relu 也写在 bf16 里形成对照。

    triton.Config({'BLOCK_M':64, 'BLOCK_N':64,  'BLOCK_K':32, 'GROUP_SIZE_M':8}, num_warps=4, num_stages=4),
    triton.Config({'BLOCK_M':128,'BLOCK_N':64,  'BLOCK_K':32, 'GROUP_SIZE_M':8}, num_warps=4, num_stages=3),
    # ... 14 configs
], key=['M','N','K'])
@triton.jit
def matmul_sub_mul_relu_kernel(...):
    # 全 fp32 accumulator, NO .to(bfloat16) / .to(float16)
    acc = tl.zeros((BLOCK_M, BLOCK_N), dtype=tl.float32)
    for k_offs in range(0, K, BLOCK_K):
        a = tl.load(A_ptr + ...)
        b = tl.load(B_ptr + ...)
        acc += tl.dot(a, b, allow_tf32=True)  # ★ TF32 但不降 dtype
    # epilogue: sub + mul + relu (all fp32)
    acc = (acc - subtract_value) * multiply_value
    acc = tl.maximum(acc, 0.0)
    tl.store(C_ptr + ..., acc)

走 fp32 + TF32 路径，没用 .to(bf16)，是 program_md 在 L1/L2 中难得的"诚实写法"。Speedup 1.88× 来自 epilogue fusion (sub+mul+relu 合并到 matmul kernel 内，省 3 个 elementwise pass)。

__global__ void fused_epilogue_kernel(
    const __nv_bfloat16* __restrict__ input,
    const float*         __restrict__ bias,
    __nv_bfloat16*       __restrict__ output,   // ★ output 还是 bf16
    float subtract_val, float multiply_val,
    int M, int N
) {
    int row = blockIdx.y, col = blockIdx.x * blockDim.x + threadIdx.x;
    if (row < M && col < N) {
        float val = __bfloat162float(input[row*N + col]);
        if (bias) val += bias[col];
        val = (val - subtract_val) * multiply_val;
        val = fmaxf(val, 0.0f);
        output[row*N + col] = __float2bfloat16(val);   // ★ 输出再塌回 bf16
    }
}

class ModelNew(nn.Module):
    def forward(self, x):
        # ★ wrapper 把 fp32 input cast 到 bf16
        x_bf = x.to(torch.bfloat16)
        w_bf = self.linear.weight.to(torch.bfloat16)

        out_bf = torch.empty_like(gemm_out)

class Model(nn.Module):
    def __init__(self, in_channels, out_channels, kernel_size, stride, padding,
                  maxpool_kernel_size, maxpool_stride, hardtanh_min, hardtanh_max):
        super().__init__()
        self.conv_transpose = nn.ConvTranspose2d(in_channels, out_channels, kernel_size,
                                                  stride=stride, padding=padding)
        self.maxpool = nn.MaxPool2d(kernel_size=maxpool_kernel_size, stride=maxpool_stride)
        self.hardtanh = nn.Hardtanh(min_val=hardtanh_min, max_val=hardtanh_max)

    def forward(self, x):
        x = self.conv_transpose(x)        # cuDNN
        x = self.maxpool(x)               # cuDNN
        x = self.hardtanh(x)              # PyTorch fused (CUDA built-in)
        x = torch.mean(x, dim=(2, 3), keepdim=True)
        x = torch.tanh(x)
        return x

整文件没有任何自定义 kernel！verified 1.07× 完全来自 program_md 的某种环境层 tweak（torch.backends 配置 / channels_last memory format / autotune cache 等）— 实测速度比 ref 微微快。这是 program_md.md "Decision Framework: PyTorch is already optimal → keep" 的字面执行，反而成为本题唯一 real_fp32。

为什么 ako4all 在 P10 输了:

  1. trajectory 太长导致 ITERATIONS.md 留痕断档: 36 iter 远超 P9 的 5 iter，LLM 倾向"快速尝试 + 跳过 hypothesis 写作"。当 hypothesis 段为空时，反 reward hacking 的核心防线失效。
  2. iter-18 选 fp16 over bf16 的决策没留痕: bf16 在 ConvTranspose 上误差更小（指数位多 1 bit），如果 LLM 在 hypothesis 写"为什么 fp16 不放大 max-pool 后的 reduce error"，可能会发现选 fp16 不安全。
  3. ConvTranspose 后接 mean reduce 的精度风险被忽视: fp16 的 ConvTranspose 输出在 [-1, 1] 量级，但 256×256 像素做 mean 时积累误差能到 1e-2 量级，恰好擦 atol=1e-2 边界。

为什么 program_md 不动反而赢:

  1. 9 题样本是"被选中的硬题": program_md 在 L2 上选题倾向"PyTorch ref 已最优"的题（P10 baseline 是 cuDNN ConvTranspose + cuDNN MaxPool + ATen Hardtanh+mean+tanh），LLM 写自定义 kernel 收益 < 5%，保留 baseline + 微调 backends 反而是上策。
  2. program_md.md Decision Framework 明写 "when PyTorch is already optimal, do not over-optimize": 这次 LLM 真的执行了这条建议。

教训:

  1. ITERATIONS.md 留痕约束不是绝对防线: 长 trajectory (>20 iter) 上 LLM 容易跳过 hypothesis 段；ako4all 应该在 prompt 加 "ITERATIONS.md 留痕是 commit 前置条件，未填禁 commit"。
  2. fp16 不是 ConvTranspose 的安全 dtype: ConvTranspose 在 fp16 下的边界 padding 算法有舍入累积，与后续 reduce 叠加会突破 atol=1e-4。即使 ConvTranspose 自身能用 fp16，应在 fp32 epilogue 收敛误差（与 §5.2 P9 同样的策略）。
  3. "do nothing" 也是合法策略: program_md 在本题的胜利是反直觉的 — 当 baseline 已经 cuDNN 最优时，写自定义 kernel 的边际收益 < dtype cast 风险。L2 上有 ~30 题属于这类，应在 prompt 加 "如果 ncu profile 显示 baseline 已达 80%+ roofline，保留 baseline"。

5.4 三例总结的 prompt 启发

教训 来自案例 落地到 prompt 的建议
LLM 权重内的代数知识是 L2 最大加速来源 §5.1 P18 加 "Before writing any kernel, check if op chain has algebraic simplifications (sum-after-matmul, identity-on-singleton-dim, distributive properties)"
