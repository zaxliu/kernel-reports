program_md / program_kb 完全失效的细分

156 个 program_md/kb 全 fake 样本中:

  • fake_logic_bug: 208 个(写错 kernel)
  • fake_precision_cheat: 95 个(内部 cast fp16 蒙混)

v3 时这两个框架真实率 ~30%,新批 0%。可能原因(待 root cause):

1. agent prompt 改动后 program_md/kb 输出质量崩盘

2. harness 切到 --max-concurrent 2 后这两个框架没拿到足够 GPU 时间产出真实 kernel

3. agent 学会用 .half() / tl.float16 "蒙混" atol 检查(精读 prompt 反作用)

fake_precision_cheat 占比 program_md/kb 38% 与 ako4all 仅 6.7% 的差距,强烈暗示这两个框架的 prompt 引入了 fp16 cast 倾向。

关键回归判断

1. 不是硬件问题 — 新批 BMC 全因 agent kernel 写错的 cpp_extension 触发 UVM hang,跟 v3 时硬件 instability 不同;uvm_disable_hmm=1 + cap=2 防御组合可靠。

2. 不是 verifier 问题 — verify_three_tier_l1l2_2026-06-04.py 与 verify_three_tier_l2.py 算法完全一致(只改路径),criterion 一致 fp32 atol/rtol=1e-4。

3. 是 agent 输出质量回归 — 156 题从 real 翻 fake,且 fake_logic_bug 占主流,说明新 batch 的 LLM(可能不同模型/prompt/temperature)输出编译过但行为错的 kernel 大幅增多。

4. program_md / program_kb 完全崩盘的子假设: 其专属 prompt 不知什么时候被改坏了,把模型推向 fp16 cast 路径 + 错误的 indexing 模式。
