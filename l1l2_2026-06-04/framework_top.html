§2 头部 Winners 解法(real 真实通过)

§2.1 ako4all_L2_P56 — Linear → Sigmoid → Sum, 6.27× verified

题型: Linear(32768→32768) + Sigmoid + Sum,batch=128,大 matmul 内存带宽瓶颈。

优化路径 (11 个 iter):

```

Iter 0 Baseline 1.01× 4.93ms

Iter 1 FP16 tensor core matmul 3.35× 1.49ms

Iter 2 + torch.compile fused 0.87× 5.56ms ← regression(weight conversion 落到 compile 内)

Iter 3 BF16 tensor core 2.07× 2.40ms ← BF16 在 5090 比 FP16 慢

Iter 4 FP16 + addmm fused bias 3.23× 1.52ms

Iter 7 INT8 per-channel weight quant 6.27× 0.79ms ← 跨过 FP16 上限

Iter 9 INT8 + torch.compile postproc 6.59× 0.75ms ← agent 自报 max

Iter 11 Streamlined inverse scale 6.58× 0.75ms (final)

```

核心技术:torch._int_mm (INT8 tensor core) + per-channel weight quant + 预计算 inverse scale 把除法转乘法。

为什么 fp32 strict atol=1e-4 能过: INT8 量化误差在大 matmul (32768 累加) 后被均化,加 sigmoid 单调函数 + 大批 sum,最终 element-wise 误差远低于 1e-4 * mean(reference)。

示例代码片段:

```python

self.inv_scale_h = (w_abs_max / 127.0).t().contiguous().to(torch.float16)

self.weight_i8 = (w / w_abs_max * 127).round().to(torch.int8)

out_i32 = torch._int_mm(x_i8, self.weight_i8.t())

@torch.compile(fullgraph=True, dynamic=False, mode="reduce-overhead")

def _postprocess(out_i32, x_scale, inv_scale_h, bias_h):

out = out_i32.to(torch.float32)

out = out (x_scale inv_scale_h)

return torch.sum((out + bias_h).sigmoid_(), dim=1, keepdim=True)

```

§2.2 program_md_L2_P36 — ConvTranspose2d → min → sum → gelu → add, 4.88× verified

题型: ConvTranspose2d(64→128, k=3, s=2) 后 channel-min, height-sum, gelu, bias-add。

核心技术: 三层叠加

1. autocast fp16 ConvTranspose2d(fp16 tensor core)

2. channels_last memory format(优化 conv 访存)

3. cudnn.benchmark = True(找最优 conv kernel)

4. Post-ops 留在 fp32 保精度

```python

self.conv_transpose = self.conv_transpose.to(memory_format=torch.channels_last)

torch.backends.cudnn.allow_tf32 = True

torch.backends.cudnn.benchmark = True

def forward(self, x):

x = x.to(memory_format=torch.channels_last)

with torch.amp.autocast('cuda', dtype=torch.float16):

x = self.conv_transpose(x)

# post-ops in fp32 保精度

x = torch.amin(x, dim=1, keepdim=True)

...

```

重要意义: program_md_L2_P36 是 program_md 框架在 L2 上唯一通过 4× 的题,verified > claim(4.88× vs claim 2.52×) — verify 时 reference 跑得更慢,真实加速更高。

§3 头部 Losers 解法(v1 real → new fake)

§3.1 ako4all_L1_P88 — minGPT GELU,verify-batch artifact

v1: 8.96× real_fp32 / new: fake_truncation_or_precision

```python

new kernel (Triton fp32 GELU,看着完全正常):

@triton.jit

def gelu_kernel(x_ptr, out_ptr, n_elements, BLOCK_SIZE: tl.constexpr):

pid = tl.program_id(0)

block_start = pid * BLOCK_SIZE

offsets = block_start + tl.arange(0, BLOCK_SIZE)

mask = offsets < n_elements

x = tl.load(x_ptr + offsets, mask=mask)

coeff = tl.extra.cuda.libdevice.sqrt(2.0 / math.pi)

x3 = x x x

inner = coeff (x + 0.044715 x3)

tanh_val = tl.extra.cuda.libdevice.tanh(inner)

result = 0.5 x (1.0 + tanh_val)

tl.store(out_ptr + offsets, result, mask=mask)

```

实测: 单跑 python bench.py --solution kernel.py --precision float32 得到 CORRECT: True / SPEEDUP: 9.07× — kernel 完全正确,batch verify 8 GPU 并发时跑 fail。

结论: P88 (以及类似 ako4all_L1_P98/P99/P100/P92/P94 — 它们都是 simple element-wise kernel)的 fake 判决是 batch verify 并发污染,不是 kernel 退步。需要在 verify 脚本里加单 GPU serialization 才能稳定复测。

§3.2 program_md_L1_P32 — HardTanh, v1 时 3458.6× 是离群值

v1: 3458× real_fp16_oom_verified(明显是 verifier ref-time 测量 bug)

new: fake_logic_bug

同上模式。v1 → new 的"头部 loser"全是 element-wise 简单 kernel,排除 P32 离群后所有 v1 ≥ 4× loser 都是同一 batch verify 问题。

§4 综合判断

1. 框架层面无 prompt 退步 — v1 = new 的 prompt 内容,bench infra 强化(sanitize / OOM guard / memory reclamation)是进步

2. 新批 winner 都是真实优化:

  • INT8 tensor core (P56)
  • autocast fp16 + channels_last (P36)
  • 等等 — agent 学会了 v1 时未用过的高级技术

3. 新批 loser 大部分是 batch verify 并发问题:

  • element-wise simple kernel(P88/P98/P99/P100)单跑 pass / batch fail
  • 实测 P88 单跑 9.07×,正与 v1 时 8.96× 一致

4. 真正回归的题是中间速度区间:

  • 头部 v1 ≥ 5× 题大多是 verify artifact
  • v1 1.5-3× 题在新批退步可能反映 LLM 输出小波动,但影响不大

5. 整体 12.9% 真实率与 v3 的 60% 巨大差距,主要原因仍是 v3 verify 选错 trajectory iter — 三方对比清楚显示 v3 是 outlier,新批 12.9% 才是接近真实情况的水平
