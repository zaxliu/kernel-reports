关键发现

A. 2 个 verify 工具 bug

Bug 1: bench.py CLI 不兼容(verify v1 → v2 修复)

初版 verify 脚本根据框架选 bench.py:

  • ako4all 用 bench/kernelbench/bench.py(--ref/--solution/--precision 接口)✅
  • program_md/kb 用各自 attempt root 的 bench.py(自包含 --kernel/--sizes 接口) ❌

但 verify 用 ako4all schema 的 args 调所有 bench.py → program_md/kb 全 args parse error → 全 status=fail → 0% 真实率。

修复: 永远用 ako4all 的 bench.py(框架无关,只 import kernel.py 与 ref 比较)。

  • v1 → v2: program_md/kb 真实数 0 → 13。

Bug 2: 8 GPU 并发污染(verify v2 → v3 修复中)

5/5 头部 loser 单跑(单 GPU 独占,串行)全部 PASS,但 batch verify(8 GPU 并发)判 fake。

B. v3 数据可信度问题

v3 (5/27) 报告 ako4all_L1 89.4% 真实率,新批 23.5% / 42.4%;但:

  • v1 (5/1) 同一框架真实率仅 76.9%
  • v3 报告自己提到 program_md_L1_P32 3458× 是 verifier ref-time bug
  • v3 时报告"+84 newly real"的 5 题头部成果(P88 8.77× / P40 7.78× / P100 7.65×)在新批全 fake_logic_bug
  • 但新批 batch fake ≠ 真 fake(并发问题)

结论: v3 89.4% 是 outlier(选 trajectory iter 时偏向 best speedup,但部分 best 是 bench fluke);v1 76.9% / new 真实数据是更可信的范围。

方法论教训(供未来 verify pipeline 借鉴)

1. verify 工具应该有 self-test — bench.py CLI schema 不兼容这种 bug,自检几个题就能发现。下次 verify 之前先 run-1-test。

2. verify 应该有 single-run baseline + parallel-run 对比模式 — 如果 单跑 ≠ 并跑,说明并发污染存在,要切换 serial 模式。

3. 不要轻信"老报告的 baseline" — v3 89.4% 是 verify trajectory 选 iter 不严谨,导致部分 fluke 被算成 best。

4. 失败 verdict 要看 exit_code / stderr — 我们之前判 fail 时没区分 args parse error / kernel runtime error / atol fail,导致 bug 1 被掩盖。

5. 5/5 抽样验证 ≠ 全数据 — 单跑 5 题 PASS 是强假设,但需要 481 题 serial verify 才确认。
