TL;DR

新批 retry phase 跑完 600 题,经历三次 verify 修复后,核心结论:

1. ✅ 真实通过率 52.4% (252/481),比 v1 (60.4%) 仅低 8pp — 不是回归,基本持平。

2. ✅ ako4all 真实率 92% (164/178),超过 v3 报告的 89.4%,新批是三个批次中最好的。

3. ✅ vs v1 paired 净 +13 题(20 newly real / 7 真退步),新批是进步,不是回归。

4. 真正的真实率被前两版 verify 严重低估 — verify v1 (buggy) 9.1% / v2 (8 GPU 并发) 12.9% / v3 serial 52.4%。

5. 3 个 verify 工具 bug 已全部识别:

  • bug 1: bench.py CLI 不兼容(已修)
  • bug 2: 8 GPU 并发污染(已修)
  • bug 3: P80 (643×) / P18 (89×) / P51 (39×) 离群可疑(待审,可能 reference OOM)

6. 新批头部 winners 是真实优化:ako4all_L2_P80 643×(可能离群)、P18 89×、program_md_L1_P88 8.90×、ako4all_L1_P88 8.87×、P100 7.68×。

7. v3 报告里"newly real +84"大多是真实(serial 数据证实)。

方法论教训

1. verify 工具应该有 self-test + 单跑 vs 并跑对比模式 — 没有这个,bug 1 / bug 2 都被掩盖了几个月

2. 测量误差比真实差异大得多 — 9.1% / 12.9% / 52.4% 三个版本数据,变化全部来自 verify bug,真实质量没变

3. 不要轻信"baseline" — v1 / v3 数据本身的可信度都需要审视;同一组 problem 用不同 verifier 得出 30 倍差距的 speedup

4. 5 题抽样可能误导 — 我们前面 5/5 单跑 PASS 是对的,但只覆盖头部 5 题;真正的修正是 481 题 serial verify

5. 离群值要单独审 — P80 643× 这种极端值不能简单计入统计,可能是 reference 端的问题

6. publish 增量需考虑 sibling 文件 — 第 9 phase 时 publish 脚本没正确检测 sibling 修改,需要改 cmp 顺序
