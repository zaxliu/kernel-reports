verify v2 修复说明

初版 verify 脚本 bug(build_html_l1l2_2026_06_04.py 同一 commit 里):

```python

错误版本(初版):

if fw == "ako4all":

cand = attempt / "bench" / "kernelbench" / "bench.py"

else:

cand = attempt / "bench.py" # ❌ program_md/kb 自己的 bench.py 用 --kernel/--sizes

```

但 verify_three_tier 一直用 --ref/--solution/--precision CLI 调 bench.py,这只兼容 ako4all 的 bench.py。program_md/kb 自包含的 bench.py 接到这套 args 直接 argparse error 退出 → status=fail → fake_logic_bug。

修正版(verify v2):

```python

永远用 ako4all 的 bench.py(框架无关 — 只 import kernel.py 与 ref 比较)

cand = attempt / "bench" / "kernelbench" / "bench.py"

if cand.exists(): return cand

program_md/kb attempt dir 没有,fallback 到任意 ako4all attempt

for ako_bench in RUNS_ROOT.glob("ako4all/L/p/a*/bench/kernelbench/bench.py"):

return ako_bench

```

这与 v3 时 verify_three_tier_l2.py 的设计一致(它也是只列 bench/kernelbench/bench.py candidates)。

失败模式分析

fake_logic_bug (267, 55%) — 编译过但 fp32 跑错

per-fw:

  • ako4all: 23+52 = 75
  • program_md: 52+41 = 93
  • program_kb: 54+45 = 99

fake_precision_cheat (97, 20%) — 内部 cast fp16/bf16

  • program_md: 17+35 = 52
  • program_kb: 11+26 = 37
  • ako4all: 0+8 = 8 (主要 ako4all_L2 出现少量,L1 完全没有 — 框架防护起作用)

→ program_md/kb 模型倾向用 fp16 cast 蒙混 atol — 它们 prompt 里没有像 ako4all TASK.md 那样明确禁止 reward hacking。

fake_truncation_or_precision (39, 8%) — fp16 通过但 fp32 不过

  • ako4all_L1: 10
  • ako4all_L2: 17
  • program_md_L1: 1
  • program_md_L2: 5
  • program_kb_L1: 2
  • program_kb_L2: 4

ako4all 占大头,但这是因为它 prompt 鼓励 fp16 IO + fp32 compute(混合精度优化),少量精度损失正常,只是被 strict atol 打掉。

fake_or_broken (16, 3%) — 编译失败/runtime crash

主要 ako4all_L1 的 16 个,都是 kernel 写错(往往 shared memory race / index 越界)被 verify run 当场 crash。

关键发现

1. 真实率 12.9% 是新批的真实水平(初版报 9.1% 是 verify bug 偏低 4pp)

2. ako4all 仍是唯一可用框架(42.4%/17.2% L1/L2),其他两个框架 1-5% 真实率

3. program_md/kb 系统性产出 fake_precision_cheat(89/97 = 92% 占该类别),说明 prompt 没禁止 fp16 cast

4. ako4all 框架 prompt 有效抑制 reward hacking(L1 完全无 fake_precision_cheat,L2 仅 8 个)

5. 新批 max 6.27×(P56),不是初版报告说的 8.80×(P66 实际 fake)

6. program_md_L2_P36 4.88× 是真正惊喜:program_md 在 L2 上居然有 4.88× 真实加速题,初版 buggy 数据完全漏掉
