失败模式分析

fake_logic_bug (303, 63%)

agent 写出了编译通过但 fp32 输出错的 kernel。最常见根因:

  • index 算错 (off-by-one)
  • 共享内存 race
  • block/thread 维度选错

per-fw 分布:

  • ako4all: 44+51 = 95
  • program_md: 54+45 = 99
  • program_kb: 58+51 = 109

fake_precision_cheat (101, 21%)

kernel 内部 cast 到 fp16/bfloat16 (tl.float16, .half(), .to(torch.float16) 等),fp32 atol=1e-4 跑不过但 claim 时是 fp16 atol=1e-2 通过的"假性能"。全部来自 program_md / program_kb:

  • program_md: L1=17 + L2=40 = 57
  • program_kb: L1=12 + L2=26 = 38
  • ako4all: L1=0 + L2=6 = 6 (新出现,L1 v3 时只有 1 个)

隔离 program_md/kb 完全失效

156 个 program_md/kb 样本全员 fake (54+45+58+51 = 208 fake_logic_bug + 17+40+12+26 = 95 fake_precision_cheat,合计 303 fake;实际有 verdict 数=156,因为部分样本 claim<1× 被跳)。这两个框架在新 batch 没有产出任何真实 fp32-correct kernel。

关键发现

1. program_md / program_kb 完全失效 — 这是新 batch 最关键的回归。Phase 3 与 v3 (各 ~30% real) 对比时需重点排查 retry phase 是不是 agent 框架升级或 prompt 改动导致。

2. ako4all 真实率从 v3 89.4% (L1) 降至 23.5% — 也是大幅回归,但仍是三框架中唯一可用。

3. fake_precision_cheat 占比 21% 远高于 v3 — agent 学会用 .half() / tl.float16 蒙混 atol 检查,需要在 verifier 或 prompt 上补防。

4. 头部 speedup 数量减少 — ≥5× 只有 2 题 (v3 是 7 题),≥2× 只有 7 题 (v3 是 36 题)。

5. 系统侧 OK — 2 次 BMC 都因 agent kernel bug 触发,非硬件;BMC 流程 + cap=2 + uvm_disable_hmm 三套防御组合可靠。
